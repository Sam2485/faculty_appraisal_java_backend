# Security & Authorization

The system implements a robust security layer using Spring Security and JWT, ensuring stateless authentication and hierarchical role-based access control.

## Authentication (JWT)

- **Stateless**: No server-side session storage. The application uses `SessionCreationPolicy.STATELESS`.
- **JWT Content**: Tokens include:
    - `sub` (Subject): User's email address.
    - `role`: The primary appraisal role assigned to the user.
    - `iat` / `exp`: Issue and expiration timestamps (default lifetime is 24 hours).
- **Verification Tokens**: Specialized tokens generated for email verification include a `purpose="email_verification"` claim. The system explicitly checks for this claim to prevent verification tokens from being used for general API authentication.
- **Lifecycle**: Tokens are generated by `JwtUtils` upon successful login and validated for every subsequent request via `JwtAuthFilter`.

## CORS Configuration

The system is configured to allow cross-origin requests from specific trusted frontends.
- **Allowed Origins**:
    - Local development: `http://localhost:5173`, `http://localhost:3000`, etc.
    - Production: `https://facultyappraisal-495011.web.app`, `https://faculty-appraisal-frontend-*.run.app`.
- **Allowed Methods**: `GET`, `POST`, `PUT`, `DELETE`, `OPTIONS`, `PATCH`.
- **Exposed Headers**: `X-Process-Time` (used for monitoring performance).
- **Credentials**: Supported (`allowCredentials(true)`).

## Authorization (RBAC)

Role-Based Access Control (RBAC) is enforced at multiple levels:

### 1. Endpoint Security

Configured in `SecurityConfig.java`, defining which paths are public and which require authentication:

- **Public Endpoints (Unauthenticated)**:
    - Authentication Flow: `/api/v1/auth/login`, `/api/v1/auth/register`, `/api/v1/auth/verify-email`, etc.
    - Announcements: `GET /api/v1/announcements`.
    - Feedback Submission: `POST /api/v1/feedback`.
    - Static Files: `/uploads/**` (serving uploaded documents in local storage mode).
    - Health Checks: `/actuator/health/**`, `/actuator/info`.
- **Authenticated Endpoints**: All other `/api/v1/**` paths require a valid JWT.

### 2. Hierarchical Authorization Logic

This is implemented in `CurrentUser.hasAuthorityOver(targetEmail, targetRole, targetDept, targetSchool)`. It uses a **Role Weight** system (defined in `RoleUtils`):

- A reviewer can only see/review a subordinate if their role weight is higher than the subordinate's role weight.
- Additional checks ensure they belong to the same Department or School (unless they are higher-level roles like Dean or VC).

## Roles and Weights

| Role | Weight | Visibility Scope |
| :--- | :--- | :--- |
| `super_admin` | 6.0 | University-wide (Full System Access) |
| `admin` / `hr` | 5.0 | University-wide (Management Access) |
| `vc` | 4.0 | University-wide |
| `registrar` | 3.5 | Non-Teaching (Broad Access) |
| `dean` | 3.0 | School Group (Engg/Non-Engg) |
| `center_head` | 2.5 | Specific Centers |
| `director` / `section_head` | 2.0 | School / Section |
| `reporting_officer` | 1.5 | Direct Subordinates |
| `hod` | 1.0 | Department |
| `faculty` / `staff` | 0.0 | Personal Only |

## Password Security

- **Hashing**: Passwords are encrypted using `BCryptPasswordEncoder` before being stored in the database.
- **Brute-Force Protection**: `LoginAttemptService` monitors failed login attempts and flags IPs or accounts that exceed a configured threshold within a time window.
